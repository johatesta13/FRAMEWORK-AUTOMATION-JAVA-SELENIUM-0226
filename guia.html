Guía básica de Git, GitHub e IntelliJ

Git — control de versiones local
Rastrea cambios en tu código. Cada commit es una foto del proyecto.
1
Instalación y configuración
Instalá desde git-scm.com, luego configurá tu identidad:
git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"
2
Iniciar un repositorio
git init           # crea repo en la carpeta actual
git clone <url>      # copia un repo existente
3
El ciclo diario
Trabajar→ git add→ git commit→ git push
git status           # ver qué cambió
git add .            # staging de todo
git add archivo.js   # staging específico
git commit -m "mensaje"  # guardar
git push origin main   # subir al remoto
Mensajes descriptivos: "Agrega validación de formulario", no "cambios".
4
Ramas (branches)
git branch                  # ver ramas
git checkout -b nueva-feature  # crear y cambiar
git merge nueva-feature        # fusionar en main
5
Comandos útiles extra
git log --oneline     # historial compacto
git pull origin main  # traer cambios del remoto
git diff              # ver diferencias sin commitear
git stash             # guardar cambios temporalmente
GitHub — repositorio remoto y colaboración
Tu código en la nube. Colaborá con Pull Requests e Issues.
1
Crear y conectar un repositorio
Creá el repo en github.com (+ → New repository), luego conectalo:
git remote add origin https://github.com/usuario/repo.git
git push -u origin main
Con -u configurás el upstream. Después solo necesitás git push.
2
Fork y Clone
Para contribuir a proyectos ajenos: hacé Fork (copia en tu cuenta), luego:
git clone https://github.com/TU-USUARIO/repo.git
3
Pull Request (PR)
Crear branch→ Cambios→ Push→ Abrir PR→ Merge
Al hacer push de una nueva rama, GitHub te muestra el botón Compare & pull request automáticamente.
4
Issues y README
Los Issues son tareas o bugs. Referencialos en commits con #número. En el PR podés escribir Closes #5 para cerrarlo automáticamente al hacer merge.

El README.md es la portada de tu repo: describí qué hace el proyecto, cómo instalarlo y cómo usarlo.
5
Autenticación con SSH
ssh-keygen -t ed25519 -C "tu@email.com"
# Pegá ~/.ssh/id_ed25519.pub en
# GitHub → Settings → SSH and GPG Keys
Con SSH no necesitás ingresar usuario y contraseña en cada push.
IntelliJ IDEA — Git integrado
Hacé casi todo sin escribir comandos. Git nativo en el IDE.
1
Clonar o abrir un proyecto
File → New → Project from Version Control para clonar. Si ya tenés el proyecto, al abrirlo IntelliJ detecta el repo Git automáticamente.
2
Commit desde el IDE
Abrí el panel Commit con Ctrl+K. Desde ahí:
  • Tildás qué archivos incluir
  • Escribís el mensaje del commit
  • Hacés Commit and Push en un solo clic
3
Branches en IntelliJ
Hacé clic en el nombre de la rama (margen inferior derecho). Desde ahí podés crear, cambiar y fusionar ramas.
Para merge: desde main, clic derecho en la otra rama → Merge into Current.
4
Historial y diferencias
Git → Show Git Log abre el historial visual. Filtrá por autor, fecha o archivo. Al hacer clic en un commit ves exactamente qué cambió (diff visual coloreado).
5
Atajos esenciales
CommitCtrl+K
PushCtrl+Shift+K
PullCtrl+T
Ver cambiosAlt+9
Historial archivoAlt+9 → Log
Resolver conflictoGit → Conflicts
En Mac reemplazá Ctrl por ⌘ en la mayoría de los atajos.
Flujo completo — de cero a producción
Cómo se usan Git, GitHub e IntelliJ juntos en un proyecto real.
1
Git Setup inicial (una sola vez)
Creás el repo en GitHub, lo clonás con IntelliJ (File → New → Project from Version Control) y configurás nombre/email en Git.
2
Git Crear una rama para cada tarea
Nunca trabajes directo en main. Para cada feature o fix:
git checkout -b feature/login
O desde IntelliJ: clic en la rama → New Branch.
3
IntelliJ Trabajar y commitear seguido
Commits pequeños y frecuentes son mejor que uno gigante al final. Desde IntelliJ: Ctrl+K → tildar archivos → escribir mensaje → Commit.
4
GitHub Push y Pull Request
Cuando terminaste: Ctrl+Shift+K para push. En GitHub aparece Compare & pull request. Describís los cambios y pedís review si trabajás en equipo.
5
Git Merge y actualización
Aprobado el PR, se hace merge en GitHub. Luego localmente:
git checkout main
git pull origin main
¡Listo! Este ciclo se repite para cada nueva funcionalidad.
